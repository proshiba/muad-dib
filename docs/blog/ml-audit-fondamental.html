v2.10.5 — Quand un changement de label fait passer le ML de 37% a 98%

2026-03-22
ml audit xgboost monitoring pipeline

3 mois de monitoring, 0 malware confirme

Le moniteur MUAD'DIB tourne depuis decembre 2025. ~10 000 packages npm scannes par jour, 24/7. Environ 600 alertes webhook Discord quotidiennes. Et apres 3 mois : zero malware confirme.

Pas parce qu'il n'y en a pas. Le dataset Datadog contient 14 587 malwares in-scope, soit environ 26 par jour en moyenne sur la periode couverte. Le probleme : les 600 alertes quotidiennes se ressemblent toutes. Des cards Discord identiques, orange, sans hierarchie. Personne ne les trie, parce que c'est humainement impossible.

L'audit fondamental

J'ai pose 20 questions sur le pipeline. 5 causes racines identifiees. La plus grave : la contamination des labels ML.

Le schema etait simple :

  1. Le moniteur detecte un package suspect (score ≥ 20)
  2. Le sandbox Docker l'execute dans un container isole
  3. Le sandbox ne detecte rien (pas de canary token exfiltre, pas d'activite reseau suspecte)
  4. Le moniteur labellise automatiquement le package comme "fp" (faux positif)

Le probleme : le sandbox n'avait pas de honey tokens pendant 3 mois. Un sandbox propre ne prouve rien. Le malware peut simplement ne pas s'etre active (time bomb, environnement CI non detecte, cible specifique). 8176 records contamines avec le label "fp" alors qu'on ne savait pas s'ils etaient vraiment FP.

Resultat : le classifier ML apprenait a dire "tout est FP". Precision 37.2%, recall 99.8%. Il classait quasi tout comme malware (recall eleve) mais la plupart de ses predictions positives etaient fausses (precision catastrophique) parce que ses donnees d'entrainement "FP" contenaient probablement des vrais malwares.

Le nettoyage

Le fix : convertir les 8176 labels "fp" automatiques en "unconfirmed" et les exclure de l'entrainement ML. Seuls les labels valides par review manuelle (manualReview: true) sont desormais acceptes comme "fp".

Avant/apres, meme code, memes features, donnees propres :

MetriqueAvant nettoyageApres nettoyage
Precision37.2%97.8%
Recall99.8%93.3%

Le recall a baisse de 99.8% a 93.3% — normal, le modele ne trichait plus en classant tout comme positif. Mais la precision est passee de 37% a 98%. Le meme algorithme, les memes hyperparametres, juste des donnees propres.

Les modeles ML

ML1 — Malware detector

XGBoost, 114 arbres de decision, 21 features, seuil de classification 0.500.

Les top features discriminantes :

  • score — le score de risque global. Logique : c'est l'aggregation de toutes les heuristiques
  • max_single_points — le poids de la finding la plus grave. Un malware a generalement au moins une finding CRITICAL
  • file_score_max — le score du fichier le plus suspect. Les malwares concentrent le code malveillant dans 1-2 fichiers
  • threat_density — nombre de findings / nombre de fichiers. Un package avec 3 findings dans 1 fichier est plus suspect que 3 findings dans 500 fichiers

Le modele est un complement aux regles, pas un remplacement. Les regles decident "quel type de menace", le ML decide "est-ce que l'ensemble de signaux ressemble a un vrai malware ou a du bruit".

ML2 — Bundler detector

XGBoost, 98 arbres, 30 features, seuil 0.100.

Probleme specifique : les bundlers (webpack, rollup, esbuild) generent du code minifie qui accumule des findings legits — eval, Function, require dynamiques, patterns d'obfuscation — qui ressemblent a du malware. ML2 detecte ces cas et reduit leur score.

F1 de 0.996. Un warning honnete : ce F1 > 0.99 est suspectement eleve. La feature unpacked_size_bytes correle fortement avec les bundlers (les bundles sont gros) et pourrait constituer un leakage. Le modele reste utile mais son F1 reel en production sera probablement inferieur.

Le triage P1/P2/P3

600 alertes identiques par jour, ca ne sert a rien. Le cerveau humain ne peut pas traiter ca. La solution : classifier visuellement les alertes.

  • P1 (rouge) — IOC match, types haute confiance (lifecycle_shell_pipe, dangerous_binary_dropper...), canary token exfiltre, sandbox positif. Ce sont les vrais malwares. ~26 par jour en moyenne.
  • P2 (orange) — Score ≥ 50, compound scoring declenche, lifecycle + intent coherence. Suspect mais pas confirme.
  • P3 (jaune) — Le reste. Score 20-49 sans signal fort. La grande majorite des alertes, et la grande majorite des faux positifs.

Ce que ca change : les ~26 vrais malwares sont visuellement distincts des ~574 alertes FP/bruit. Un humain peut scanner les P1 en 2 minutes par jour au lieu de noyer dans 600 cards orange.

Investigation manuelle : confirmation que ca ne scale pas

J'ai investigue manuellement 9 packages live signales par le moniteur. Resultat : tous FP. Des bundlers, des fichiers de test, des CLI tools avec child_process legitimes.

Temps moyen par investigation : 5-10 minutes par package. A 26 vrais malwares + 574 FP par jour, le triage humain exhaustif prendrait 50+ heures/jour. Impossible.

La suite : LLM Phase 3

Le triage P1/P2/P3 resout le probleme visuel, mais pas le probleme de volume. Il faut un filtre automatique entre P2/P3 et la review humaine.

Le plan : Claude 3.5 Haiku en shadow mode pour analyser les alertes P2/P3. Le LLM recoit le contexte complet (findings, code, metadata) et decide : vrai positif probable, faux positif probable, ou incertain. Cout estime : ~$17/mois pour le volume actuel.

Phases prevues :

  1. Shadow mode — le LLM tourne en parallele, ses decisions sont loguees mais n'affectent pas le pipeline. On mesure l'accord avec le triage humain.
  2. Enforce P3 — le LLM filtre les alertes P3 (jaunes). Si le LLM dit "FP probable", pas de webhook.
  3. Enforce P2+P3 — extension aux alertes P2 (orange).

Les P1 ne seront jamais filtrees par le LLM. Un IOC match ou un canary token exfiltre doit toujours declencher une alerte.

Comparaison industrie

MUAD'DIB n'est pas seul dans ce domaine :

  • Socket (USA, $65M funding) — ML behavioral analysis, cloud sandboxing, CI/CD integrations
  • Phylum (rachete par Veracode) — Automated risk analysis, CI blocking
  • Snyk (UK/Israel) — Vulnerability database massive, ML triage
  • Semgrep Multimodal — Analysis multi-langage, rules community
  • OSCAR (Chine) — Academic supply-chain analysis

MUAD'DIB est le seul acteur francais dans ce domaine. Un projet open source educatif qui ne remplacera jamais les outils enterprise, mais qui prouve qu'un dev solo peut avoir un impact reel avec des donnees propres et des metriques honnetes.
